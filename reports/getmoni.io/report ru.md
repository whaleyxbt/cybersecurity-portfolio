# Vulnerability Report: GCS Bucket Compromise via Broken Access Control & Unrestricted File Upload

**Target:** getmoni.io / API & Storage Infrastructure
**Severity:** High (P2)
**CVSS (Common Vulnerability Scoring System Version 4.0):** 7.9 (High)
**Author:** whaleyxbt
**Date:** 10.04.2026

## 1. Executive Summary

В ходе исследования безопасности инфраструктуры проекта была выявлена цепочка критических уязвимостей, связанная с некорректной настройкой прав доступа к облачному хранилищу (Google Cloud Storage) и отсутствием валидации при загрузке файлов через API. 

Данные уязвимости позволяют неавторизованному злоумышленнику просматривать конфиденциальные файлы проекта (включая приватные списки кошельков и PII сотрудников), а также использовать инфраструктуру компании в качестве бесплатного хостинга для произвольного (в том числе вредоносного) контента.

## 2. Vulnerabilities Description

### 2.1. Broken Access Control (Unauthenticated Bucket Listing)

Базовая уязвимость заключается в открытом доступе к листингу директорий проектного GCS-бакета (`<REDACTED_BUCKET>`).

* **URL:** `<REDACTED_GCS_BUCKET_URL>`
* **Описание:** Любой неавторизованный пользователь может получить полный XML-список всех объектов в хранилище. Права доступа настроены некорректно: вероятно, выдано разрешение `storage.objects.list` для группы `allUsers`. Это полностью компрометирует структуру файлов компании.

### 2.2. Unrestricted File Upload (Bypassing Intended Use)

* **Эндпоинт:** `POST <REDACTED_UPLOAD_ENDPOINT>`
* **Описание:** Эндпоинт позволяет загружать изображения в бакет без какой-либо авторизации (API публично доступно через Swagger). Хотя сервер ограничивает типы файлов изображениями и добавляет случайный суффикс (предотвращая прямую перезапись), отсутствие rate-лимитов и проверки прав позволяет злоумышленнику автоматизированно загружать неограниченное количество тяжелых медиафайлов. Это ведет к исчерпанию ресурсов (Storage DoS) и финансовым убыткам для компании за счет оплаты облачного хранилища.
### 2.3. Sensitive Data Exposure & Improper Assets Management

Благодаря открытому листингу (п. 2.1), была обнаружена утечка критических данных:

1.  **Business Logic Data:** В приватной директории с проектными данными находится файл со списком EVM-адресов. Раскрытие этих данных компрометирует приватность пользователей проекта и создает векторы для таргетированного фишинга.

2.  **PII (Personal Identifiable Information):** В непроизводственной директории с личными данными были доступны файлы, связанные с сотрудниками.

Точные пути, имена, даты, места и другие персональные детали намеренно скрыты в публичной версии отчёта.

Использование корпоративных хранилищ в личных целях (Improper Assets Management) ведет к серьезным репутационным рискам и рискам раскрытия приватности сотрудников.

## 3. Business Impact

Комбинация данных уязвимостей несет следующие риски для проекта:

* **Phishing & XSS:** Вероятно, возможна загрузка произвольных файлов (`.html`), что позволяет реализовать Stored XSS в контексте доверенного домена хранилища и хостить фишинговые страницы за счет ресурсов команды moni.

* **Resource Exhaustion (Storage DoS):** Отсутствие авторизации и лимитов на эндпоинте загрузки позволяет злоумышленнику автоматизированно забить бакет мусорными данными. Это приведет к значительным финансовым потерям за счет оплаты счетов Google Cloud (хранение и трафик) и отказу в обслуживании (DoS).

* **Reputational & Privacy Damage:** Публичная утечка PII и приватных вайтлистов.

## 4. Steps to Reproduce (Proof of Concept)

**Шаг 1. Эксплуатация Unauthenticated Listing:**

Перейдите по скрытой ссылке GCS-бакета. В ответе будет возвращен XML со структурой всего хранилища.

**Шаг 2. Доступ к конфиденциальным активам:**

Используя ключи из листинга, можно обратиться к внутренним файлам напрямую:
* Списки кошельков: `<REDACTED_WALLET_LIST_OBJECT>`
* Личные файлы: `<REDACTED_PERSONAL_DATA_OBJECTS>`
* `<REDACTED_PERSONAL_DATA_INDEX_OBJECT>`

**Шаг 3. Эксплуатация Unauthenticated Image Upload:** 

Перейдите к скрытой публичной документации API и используйте затронутый upload-эндпоинт.

## 5. Remediation (Рекомендации по устранению)

1. **Настройка политик IAM GCS:** Снять права `storage.objects.list` для `allUsers` на уровне затронутого бакета. Доступ должен быть гранулярным.
    
2. **Защита эндпоинтов загрузки:** Внедрить обязательную проверку токенов авторизации для затронутого upload-эндпоинта.
    
3. **Аудит хранилища:** Провести ревизию бакета, удалить личные данные сотрудников и перенести файлы бизнес-логики в закрытое хранилище.
